Awesome Code Katas

I've been doing quite a few code katas, especially at Dallas Hack Club. Here is a list of code katas that you may want to try out. I recommend tackling them in the order they are listed :-).

Perfect Code

It's important that throughout the implementation of any kata your code should be perfect. A kata will help you find your perception of perfect code. The journey is more important than the destination. Take your time with the kata and at any point through the kata make sure the code you have written is perfect by your standards (even if the implementation is naive).

  • TOC
  • Prime Factors - Rite of Passage
  • Bowling Kata - Reinforcement
  • Mars Rover - The Transition
  • Poker Kata - Picking the Right Test
  • Vending Machine Kata - Exploring Behaviors
  • Map Kata - En Passe
  • Conway's Game of Life
  • Mancala

What Makes a Good Kata?

The links above are what I consider good katas. What follows are some characteristics of a good kata, you don't have to read the rest of this page to get started with code katas. If you want the motivation behind the katas above, you'll find the explanation below helpful.

Experience

I've been doing code katas for over a year now, and can honestly say that they have made me a better developer. Here are a few characteristics I've noticed about good code katas. Also keep in mind that good code katas emphasize only a few things on this list at a time (i.e. a code kata that tries to touch on all these things at once may not be that great).

The Toy Kata - Easy Test Progression

A good introductory kata for people new to TDD should have the characteristic of easy test progression. After giving a hint for the first test, the next tests should come naturally. [The Prime Factors Kata] is a good example of this. The first test is easy to specify and subsequent tests follow the same mold. The intro katas serve to help developers figure out their test environment and present problems that aren't intimidating. They serve to lay down some good practices: start with the failing test, write just enough code to make it pass, don't over think the problem, avoid the impulse to just immediately solve it. Other katas that fall into this category: [Check Out Kata], [Bowling Kata], [Roman Numeral Kata], [Word Wrap Kata], [The Mars Rover Kata]. You'll eventually outgrow toy katas, but they are a very important part of learning TDD.

The Naive

A valuable lesson a kata teaches you is how to keep things simple. We as developers have preconceived notions on how a given problem should be solved. It's really great when you come across a kata that shows you how far you can take a "naive" solution (what you really learn is what you consider naive is actually just simple and sublime). The Vending Machine Kata exhibits this characteristic. Stick with the naive and you'll be surprised how far it takes you.

The En-Pass√©

There are katas out there that teach you how to deal with unexpected/new requirements. There is a fine line between simple/sublime and hackish/haphazard. Katas that exhibit this quality will let you get away with the naive implementation for a long time, but at the end they throw you a curve ball that may have you rewriting the entire thing. The "oh crap, I have to rewite this thing" moment is called an en-passe. You are meant to hit the en-passe and have to figure out how to work through it, The Map Kata exhibits this characteristic.

Picking the Right First Test, and the Right Next Test

Approaching a complex problem requires some thought on which test to write first. Katas that emphasis the right test teach the developer to break a problem apart and incrementally introduce more complex behaviors. The Conways Game of Life Kata, [Poker Kata] and Mancala Kata (to a much greater extent) exhibit this behavior. These kinds of katas are not meant to be finished in one session, the goal is to show how a complex problem can be broken into little pieces.
